# Hackathon alignment — OpenEnv Apr 2026

This document maps the project to the **22 rules** of the OpenEnv Hackathon
self-serve guide and to the **Theme #4 — Self-Improvement** brief.

## TL;DR

- **Theme:** #4 Self-Improvement
- **Environment:** [`src/openenv/environment.py`](src/openenv/environment.py)
  wraps an adaptive-curriculum math arena in which the same LLM plays
  the role of **Proposer** and **Solver** per episode.
- **Deployment:** FastAPI server ([`src/openenv/server.py`](src/openenv/server.py))
  + Dockerfile for HF Spaces under [`deployment/`](deployment/).
- **Training stack:** in-process single-GPU PPO
  ([`src/rl/ppo_trainer.py`](src/rl/ppo_trainer.py)) on top of a 1.5B
  Qwen math model, with **Qwen2.5-Math-PRM-7B** ([`src/rl/prm_scorer.py`](src/rl/prm_scorer.py))
  in 4-bit providing per-step correctness probabilities on self-play
  rollouts, and a 30 % slice of each rollout batch **grounded** on
  real GSM8K problems scored against the gold final answer. Ready to
  swap for TRL/GRPO (see §11 below).
- **Reward:** 6 independent signals — **PRM step scores (mean/min/final)**,
  **SymPy step verification**, **format compliance**, **question
  quality** (topic/clarity/novelty), **GSM8K ground-truth match** on
  grounded rollouts, and an **expert-panel curriculum modifier**.
  Combined in [`src/rl/math_environment_curriculum.py`](src/rl/math_environment_curriculum.py)
  (primary path) with [`src/rl/consensus_reward_calculator.py`](src/rl/consensus_reward_calculator.py)
  as the `--no-prm` fallback and [`src/rl/expert_panel.py`](src/rl/expert_panel.py)
  for phased modifiers.
- **Demo:** [`scripts/demo_before_after.py`](scripts/demo_before_after.py)
  renders baseline-vs-trained accuracy on GSM8K-style problems.

## Theme #4 fit — Self-Improvement

> *Create environments where agents can learn to generate new
> challenges, escalate difficulty, and improve through self-play or
> adaptive curricula. Rather than optimizing fixed tasks, the goal is
> for agents to learn to drive their own capability growth.*

The architecture wires three ingredients together:

1. **Proposer role.** Given a curriculum-selected *instruction*
   (topic + target difficulty), the policy proposes a new math problem.
   Instructions are generated by
   [`CurriculumManager.generate_instruction`](src/rl/curriculum_manager.py)
   from topic templates parameterized by `steps`, `context`, and
   `difficulty`. This makes every episode a *new challenge*, not a
   replay of a fixed set.
2. **Solver role.** The same policy solves its own proposal. A
   **triple verifier** samples two alternate solutions and takes a
   majority vote — a form of self-play where the model checks itself.
3. **Adaptive curriculum.** Per-topic success rates drive a
   Zone-of-Proximal-Development controller
   ([`src/self_play/difficulty_controller.py`](src/self_play/difficulty_controller.py))
   that escalates mastered topics and de-escalates struggling ones.
   Topic-selection probabilities are renormalized each iteration so
   the agent spends most of its budget in the learning band
   (`success_rate ∈ [0.4, 0.7]`). This is the *recursive skill
   amplification* loop.

See [`src/self_play/arena.py`](src/self_play/arena.py) for the named
`ProposerSolverArena` that exposes this loop as a single method:

```python
arena = ProposerSolverArena(curriculum_env)
result = arena.play_episode()  # proposer_reward + solver_reward + ...
```

## Proof-of-concept ladder (ties into rules #3, #11, #14)

The project is a deliberate **capability ladder**, not an "as big as
possible" training run:

| Stage | Model | Technique | Hardware | Goal |
|-------|-------|-----------|----------|------|
| 1 — where we are | 1.5B Qwen-Math | PPO (single GPU, in-process rollouts) | 1× A100 | Prove reward curve goes up and GSM8K eval improves |
| 2 — next | 7B Qwen-Math | **PPO + LoRA** on both policy and critic | 1× A100-80 | Prove technique scales |
| 3 — production | 7B–13B | **GRPO** (drop critic) + **vLLM** rollouts + FSDP | 2–4× A100 | Ship with the full hackathon stack |

The 1.5B PPO is *not* the final artifact — it's a gate. If the reward
curve rises and eval beats SFT baseline by ≥2 pp on GSM8K, we promote
to Stage 2. If it doesn't, PPO is falsified for this domain and we
pivot before spending 7B-scale compute.

## Known gaps and planned next steps

1. **Port training to `trl.GRPOTrainer`** (rule #11 target).
   The reward function is already in the right shape. The migration
   deletes the ValueHead (~50% memory) and changes advantage
   computation to group-mean-baseline.
   Effort: ~1 day.

2. **vLLM rollout engine** (rule #12 target).
   Replace HF `.generate()` with `vllm.LLM` for ~5–10× faster rollouts
   at 1.5B, ~10–20× at 7B. Needs weight-sync hook between trainer and
   vLLM engine; the FastAPI server already exposes a clean seam for
   this.
   Effort: ~2 days.

3. **Unsloth integration** at the 7B scale-up (rule #10 target).
   Current 1.5B single-GPU footprint (~6 GB) doesn't benefit from
   Unsloth; adding it at 7B saves ~30 % memory + ~2× throughput on a
   single A100. Planned for Stage 2 of the ladder.

4. **Weight-shared proposer/solver roles** — the project already uses
   the same policy weights in both roles (implicit self-play). A
   future experiment: split the LoRA into `proposer_lora` +
   `solver_lora` and train them with different objectives
   (proposer rewarded for hitting ZPD, solver for correctness). Would
   make the self-play dimension explicit enough to publish as a
   technique report.
